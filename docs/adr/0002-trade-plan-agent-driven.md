# 交易计划由 agent 基于完整上下文生成并复核

交易计划不是由 Python 规则引擎自动生成的买卖指令，也不是凭空创建的空计划；它由 agent 基于个股 ResearchSnapshot、大盘 MarketSnapshot、每日复盘 DailyReviewSnapshot、板块资金流向、市场成交量、板块热度、个人 PortfolioSnapshot（持仓、现金、流水）和风险约束生成草案。Climbing Python 端只负责计划状态机、版本化、审计日志、触发器检测和 agent 调度。所有计划变更必须由用户手动确认，并形成新的 `plan_version`。计划与真实持仓弱联动：仅作提醒，不自动交易，执行后由用户手动录入交易流水，系统再将流水关联到对应计划版本。

## 状态

accepted

## 考虑的替代方案

1. **Python 规则引擎自动生成交易信号**：rejected。容易鼓励过度交易，且无法整合研报、公告、市场情绪等复杂上下文。
2. **允许用户直接新建空计划再补分析**：rejected。空计划缺乏证据链，违背「每个结论必须附带证据」的核心约定。
3. **计划触发后自动写入持仓**：rejected。系统边界明确不做自动交易，必须保持计划与真实持仓的隔离。

## 后果

- 需要为 agent 设计交易计划生成 prompt，明确输入上下文和输出 schema。
- Python 端需要实现计划状态机、触发器检测、版本对比和复核流程。
- 复盘模块需要计算计划收益率、执行偏差、纪律性评分。
- 未来可演进为多 agent 讨论、策略模板、回测验证，但第一版必须守住「agent 分析 + Python 工作流」的边界。
