# 持仓管理与交易计划追踪系统 PRD

## 项目结论

基于你已经完成的数据可获取性审计，这个项目**完全适合做成“个人投研研究与追踪分析系统”**，但**不应该定位成交易执行系统，更不应该在没有牌照和适当性体系的前提下定位成面对外部用户的智能荐股系统**。原因很简单：你现在能稳定拿到的，是公司公告、财务三表、业务分部、股东信息、业绩预测、日线行情、公告 PDF、宏观指标、公开市场统计、新闻舆情这类“研究层”和“跟踪层”数据；你缺的是 Level 2、逐笔成交、盘口、主力资金、券商账户同步、自动下单这些“执行层”和“微观结构层”数据。与此同时，中国证监会对证券投资咨询、投资顾问和投资者适当性有明确制度安排，也已经对误导性宣传、未登记人员提供投资建议、适当性管理不当等行为实施过处罚。换句话说，**正确的产品形态不是“替你下决定”，而是“把假设、证据、计划、偏离和复盘组织起来”**。citeturn17view0turn7search0turn0search8

因此，最合理的产品定义是：**一个面向个人投资者、围绕 A 股研究与持仓跟踪的“研究操作系统”**。它接受股票代码、持仓数据和你的交易假设，持续把公开披露信息、结构化财务数据、行情数据、市场统计和宏观数据整理成四个输出：个股全景分析、持仓状态、交易计划状态、市场温度月报。这个方向既匹配你现有数据能力，也天然规避了“没有执行能力却强行做交易系统”的方向性错误。citeturn11view0turn15search1turn11view5

## 产品定位与边界

这个产品的核心用户其实只有一个：**你自己**。因此 PRD 不应该从“多角色平台”出发，而应该从“单用户、高解释性、高可追溯性”出发。它的第一原则不是信息越多越好，而是**把研究结论变成可执行、可回看、可修正的对象**。也就是说，你输入一个股票，系统不能只输出一篇像卖方研报一样的长文；它必须同时生成三样东西：第一，当前结论；第二，结论成立的证据；第三，什么情况出现时结论应当被推翻或修正。这个“假设—证据—触发器—修正”的闭环，才是整个系统的产品核心。

边界也要定得很硬。**做研究，不做交易执行；做计划和追踪，不做自动下单；做条件触发和假设校验，不做盘中微观结构博弈；做面对个人的研究辅助，不做面对第三方的个性化投顾服务。** 这样划边界并不是保守，而是顺着监管和数据现实来设计。证监会现行规则要求从事证券投资咨询服务业务具备相应行政许可条件；适当性管理办法要求对普通投资者的风险承受能力进行识别和匹配；监管实践也已经明确，误导性营销、用直播或非合规人员给出投资建议，会触发处罚。你的系统如果只服务你本人，这些约束主要转化成**产品表达方式**；如果未来要商业化，这些约束会转化成**准入门槛**。citeturn17view0turn11view3turn0search8turn0search12

因此，系统的文案和输出风格要从第一天就定型成“研究工具”而不是“荐股工具”。最终页面上不要出现“建议立刻买入”“短期必涨”“胜率 90%”这类语义，而要写成“当前假设”“关键催化”“失效条件”“乐观/中性/悲观情景”“估值区间”“计划动作建议”。这既更适合新手用户，也更符合真实投资研究的工作流。

## 数据蓝图与更新机制

你的数据体系应当分成四层。**第一层是官方披露层**，这是整个系统的事实底座。上市公司年度报告必须在会计年度结束后四个月内在证券交易所网站和符合证监会规定条件的媒体披露；巨潮资讯网是中国证监会指定的深市上市公司法定信息披露网站，且其数据服务平台明确覆盖上市公司、基金、债券、公告等内容。对于 A 股个股分析模块，**财报、公告、股东变动、定期报告、临时公告**都应归入这一层，而且权重最高。citeturn11view0turn15search1turn15search7turn8search0

**第二层是结构化市场数据层**，用来做日更和量化计算。按你提供的审计结果，`stock_finance_data` 已经足够覆盖日线行情、财务三表、业务分部、盈利预测、股东信息、财务指标、公告索引、部分实时行情与技术指标；这可以作为系统的主工作数据源。它之外，公开官方源仍应并行保留：证监会官网持续发布证券市场快报和证券市场月报；上交所和深交所公开融资融券数据；国证指数/中证体系可提供指数点位、指数市盈率、样本股、行业市盈率、ESG 等数据服务，但通常需要签署数据服务协议，部分数据涉及费用。这意味着你的系统架构应该把“内部已验证数据源”作为**主供给**，把官方站点和数据服务作为**校验与补充供给**。citeturn11view5turn2search0turn12view6turn11view8

**第三层是宏观与市场情绪层**。国家统计局有明确的统计信息发布日程，并持续发布 GDP、工业增加值、社零、固定资产投资、CPI、PPI、房地产等月度/季度数据；中国人民银行公开“社会融资规模”与“货币统计概览”体系；世界银行指标 API 提供超过 45 个数据库、近 16,000 条时间序列指标，适合做全球宏观对照或长期横向比较。你的“市场和宏观月报”如果只依赖新闻，结论会很飘；如果以国家统计局、人民银行、证监会、交易所和指数公司数据为主，再用新闻做解释层，月报才会稳定。citeturn0search14turn11view10turn11view11turn12view0

**第四层是辅助情报层**，包括卖方研报、新闻舆情、社交媒体、Yahoo Finance、学术和企业工商信息。这一层只应该作为“补充证据”，不能作为最终事实来源。尤其是 Yahoo Finance，其帮助文档明确写明：所提供的信息仅供参考，不构成交易或投资用途，且不得再分发。因此它适合做补充行情、海外可比公司、分析师推荐摘要，不适合做系统中的权威主数据。citeturn12view2turn12view3

基于这四层数据，系统的更新节奏应当设计成三种任务。**日更任务**负责收盘后一键刷新持仓、价格、浮盈亏、技术状态和触发器；**事件驱动任务**负责新公告、财报、股东变动、异常价格行为出现时重算假设；**月更任务**负责宏观和市场状态报告。这个节奏能够和官方数据发布节奏对齐，也和你的使用场景吻合。citeturn11view5turn0search14turn11view10

## 核心功能需求

系统应当围绕四个主模块来设计，而且这四个模块不是并列菜单，而是一条链路。

**个股全景分析与估值报告**是入口模块。你输入股票代码后，系统要先生成一份“研究对象档案”，然后再生成“研究结论”。档案部分包括：主营业务、业务分部、行业位置、股东结构、近年财务质量、盈利预测、公告日历、关键事件。结论部分必须固定包含：投资逻辑、核心变量、财务质量判断、经营周期位置、估值区间、技术位置、催化剂、风险点、当前操作建议。这里的关键不是让模型多写，而是要把报告拆成**可打分、可比较、可复用**的字段。比如“估值结论”必须输出区间和依据，而不是一句“低估”；“风险”必须写成可观测变量，而不是空泛地说“行业竞争加剧”。因为上市公司定期报告和公告本来就是交易所网站和指定媒体的正式披露来源，系统应当默认**财报与公告优先于新闻和二手解读**。citeturn11view0turn15search7turn8search0

**持仓管理**是整个系统的日常操作中心。由于你当前拿不到券商自动同步和本地资金接口，这个模块必须把“手动导入”做成一等公民，而不是临时替代方案。持仓数据结构至少要包含：证券代码、买入日期、买入价格、持股数量、交易费用、现金账户、分红、补仓/减仓记录。系统收盘后点击一次即可完成：最新市值、单票浮盈亏、组合浮盈亏、已实现收益、总收益率、仓位结构、行业暴露、集中度、回撤、波动率、Beta、相关性、VaR。这里建议同时保留两套收益口径：**资金加权收益**用于回答“我赚了多少钱”，**时间加权收益**用于回答“我的决策质量怎样”。这会比单一收益率更真实。按你提供的审计结果，组合风险指标完全可以基于已有价格数据和 Python 自行计算，因此该模块是 P0，而不是 P1。  

**交易计划创建与跟踪优化**是本产品真正有差异化的地方。新手最常见的问题不是不会看图，而是**没有把“为什么买、买错怎么办、什么时候复盘”写清楚**。因此一个交易计划对象必须是结构化的，而不是备注文本。建议字段固定为：计划名称、标的、时间窗口、方向、研究结论版本、入场逻辑、目标价区间、分批策略、仓位上限、风险预算、失效条件、替代情景、触发器、复盘频率、人工备注。计划创建后不是静态文档，而是一个状态机：`草稿`、`激活`、`部分触发`、`完全触发`、`假设被破坏`、`关闭`、`复盘完成`。当价格突破关键位、公告与原假设冲突、财报不及预期、盈利预测显著下修、技术趋势转坏时，系统不应该直接给出“买卖命令”，而应该弹出“计划失效/需要修订”的证据面板，让你手动确认并留痕。这种设计既更专业，也更符合合规边界。citeturn17view0turn0search8

**市场与宏观月报**是你的顶层视角模块。它不能写成新闻摘要，而要写成“因子面板 + 结论摘要”。建议固定四栏：增长、通胀、流动性、市场结构。增长看工业、消费、投资、出口；通胀看 CPI/PPI；流动性看 M2、社融、利率环境；市场结构看主要指数、成交额、估值、融资融券、行业轮动、宽度。证监会有持续更新的证券市场月报和周度快报，国家统计局有稳定的统计发布日程，人民银行有社融与货币统计口径，交易所和指数公司又提供市场结构数据，所以这个模块的数据稳定性其实很高。月报的终极输出只需要三个标签：**过热、理智、悲观**，但标签后面必须展开出证据链。citeturn11view5turn0search14turn11view10turn11view11turn11view8

## 分析引擎与决策框架

这个系统最重要的不是大模型写作，而是**规则引擎与证据组织**。我建议你把分析引擎拆成三个评分器和一个状态机。

第一个评分器是**个股研究评分器**。它输出的不是“买/卖”，而是“研究质量足不足以下计划”。建议总分 100，分成五段：基本面质量 30，经营趋势与预期差 20，估值吸引力 20，技术与资金行为 15，风险暴露 15。基本面质量主要读 ROE、毛利率、净利率、经营现金流、资产负债结构、分部业务变化；经营趋势与预期差主要看过去几个报告期的边际变化和一致预期差值；估值至少同时给出相对估值和区间估值；技术面只作为交易计划辅助，而不是覆盖基本面判断。最终只做三类结论：**可跟踪、可计划、不可碰**。这样更适合新手，也避免“分数很高但不知道该做什么”的问题。

第二个评分器是**市场温度评分器**。温度判断应该做成一个可解释的合成分数，而不是让模型自由发挥。建议使用六个维度：估值分位、成交热度、融资融券扩张速度、市场宽度、波动环境、流动性环境。估值分位可来自指数公司提供的指数市盈率/行业市盈率数据；成交热度可参考交易所公开市场数据；融资融券可来自上交所和深交所公开栏目；流动性则由人民银行和国家统计局口径补充。比如温度分数高但流动性收紧，结论就不是“全面乐观”，而是“偏热，且脆弱”。这一点对新手尤其重要，因为它把“市场强”与“赔率高”区分开了。上交所公开的股票成交概况已经包含成交金额、平均市盈率等市场概况字段，融资融券数据也由交易所持续发布，指数公司则提供指数市盈率和行业市盈率等数据服务，这些都足够支撑一个稳定的温度框架。citeturn4search0turn2search0turn12view6turn11view8

第三个评分器是**计划偏离评分器**。它只回答一件事：当前行情相对原计划是在变好、没变、还是变坏。它不重新做全套研究，而是围绕交易计划中已经定义好的几个字段检查偏离：价格触发是否满足、趋势是否逆转、财报是否支持、公告是否改变中期逻辑、仓位是否越界、组合风险是否超额。如果偏离轻微，系统只更新说明；如果偏离中等，系统要求你审核；如果偏离严重，计划状态自动改为“假设被破坏”，并要求重新生成分析报告。这会让系统真正形成闭环，而不是只有分析没有执行纪律。

最后那个状态机是**研究对象生命周期**。一个股票不应该永远只有“关注”这个状态。建议状态依次为：`候选`、`研究中`、`已立计划`、`持仓中`、`计划失效`、`观察修复`、`归档`。这使得你的 watchlist、持仓和复盘库最终会统一起来，而不会变成三个互相脱节的页面。

## 技术架构、路线图与验收

技术上，这个系统最适合做成**本地优先、批处理优先、事件补充触发**的架构。原因不是技术偏好，而是你的数据现实。既然没有券商实时同步和 Level 2，追求毫秒级实时没有意义；真正关键的是**每天收盘后的稳定更新**和**事件发生后的可追溯重算**。推荐的底层对象模型只有八类：证券主数据、财务事实表、价格与指标表、公告与文本表、持仓与交易流水、交易计划、宏观时间序列、报告快照。生成式模型不要直接碰原始表，而应当只读取“已清洗、已对齐、已写明来源和时点”的中间层视图，否则很容易把不同日期、不同口径的数据混在一起。

数据链路建议是：`采集 → 标准化 → 版本化 → 特征计算 → 规则引擎 → 报告生成 → 人工修订 → 快照归档`。这里有两个关键要求。第一，**每次报告都必须带版本号和证据链接**，因为当计划被调整时，你需要知道到底是模型变了、数据变了，还是你自己的判断变了。第二，**所有手动调整都必须留痕**，因为交易计划的价值不在于第一次写得多漂亮，而在于几个月后你能回看自己当初到底错在哪里。

如果把交付拆成阶段，我建议这样定：

| 阶段 | 目标 | 交付内容 | 优先级 |
|---|---|---|---|
| 本地研究 MVP | 先做出完整闭环 | 个股全景分析、持仓导入、收盘更新、交易计划对象、告警中心、报告快照 | P0 |
| 研究增强版 | 提高分析深度 | 公告摘要、财报变化解释、估值情景、组合风险仪表盘、计划复盘页 | P1 |
| 市场月报版 | 补齐顶层视角 | 宏观月报、市场温度、行业轮动面板、宽基指数比较 | P1 |
| 外部数据增强版 | 扩展边界 | 更稳定研报接入、更多可比公司映射、企业关系图谱、可选 ESG/专利模块 | P2 |
| 实时交易化扩展 | 暂不建议现在做 | 券商 API、盘口、逐笔、自动执行、盘中策略 | 暂缓 |

验收标准不要写成“感觉好用”，而要写成硬指标。一个可接受的 P0 系统，至少应满足：收盘后一键更新成功率高且具备重跑能力；任一持仓的当日收益、累计收益和交易流水可互相校验；任一交易计划都能追溯到对应研究报告版本；任一告警都必须能展开到具体触发条件和证据；月报必须在主要宏观数据发布后一个工作日内自动出稿；任何来自 Yahoo Finance 一类受限制源的数据都不得被当作权威口径再分发，任何需要协议授权的数据都必须在系统的 source metadata 中保留“来源、时间戳、授权状态、可再分发状态”。这是因为官方和商业数据源的授权边界并不相同，Yahoo 明确限制再分发，国证指数等数据服务也明确以协议方式提供，部分涉及费用。citeturn12view2turn12view3turn11view8

最后给一个直接的产品判断：**你现在最该做的不是“更多数据”，而是“把研究、持仓、计划、复盘合成一个对象模型”**。数据层面，你已经有足够高的可行性做出一个非常实用的个人系统；真正决定成败的是，系统是否能让每一次研究结论都沉淀成计划，每一次计划变化都留下原因，每一次收益波动都能追溯到原始假设。只要这条主线成立，这个系统就会真正帮助一个新手做投资决策；否则它只会变成另一个信息聚合器。
